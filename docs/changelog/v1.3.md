# v1.3

## ✨ New Feature: Combat Outfits

Automatically swap to a combat outfit when V enters combat or a dangerous zone (shoot-on-sight areas), and restore the previous outfit when leaving.

- **Basic usage** — name an Equipment-EX outfit with the keyword `combat` (e.g. "combat", "combat gear") and it applies everywhere during combat or dangerous zones
- **Context-aware combat** — combine `combat` with context keywords for area-specific combat gear:
  - `combat street` → only in street areas
  - `combat wild` → only in wild/badlands areas
  - `combat street club` → only at street clubs (e.g. Afterlife) — highest context score wins
- **Location-aware combat** — combine `combat` with location keywords:
  - `combat badlands` → only in Badlands
  - `combat japantown` → only in Japantown
- **Dangerous zones** — entering a hostile area (e.g. gang hideouts, enemy territory) triggers combat outfits automatically, even before combat starts. If V enters actual combat while in a dangerous zone, the outfit stays. On zone exit, the pre-zone outfit is restored
- **Priority**: combat + location match → combat + context match → plain `combat` fallback
- **Restoration**: on combat/zone exit, the pre-combat outfit is restored. If invalid, the normal location/context/outdoor fallback chain runs
- **Guards**: combat outfits are skipped at home and during ripperdoc sessions. Location/context outfit changes are suppressed while in combat or dangerous zones
- **Isolation**: combat outfits never leak into normal outfit matching, and normal outfits never appear during combat resolution

## 🐛 Bug Fix: Scene Tier Location Guard

Dynamic Wardrobe now blocks all location-based outfit changes during high scene tiers (Tier3+), preventing unwanted outfit swaps during braindances, cutscenes, ripperdoc sessions, showers, and any other scripted sequence. Tier2 (walkable scripted areas like shops and clinics) remains unblocked so location outfits can apply naturally.

- **Problem** — braindances and other scene transitions involve loading screens that wipe all runtime `@addField` values. Any runtime flag (like "block location changes") gets reset mid-scene, causing the location callback and `OnGameAttached` to fire freely — swapping V's outfit mid-scene
- **Root cause fix** — uses a **persisted quest fact** (`dw_block_location`) instead of runtime fields. The `OnSceneTierChange` callback sets the flag when the game enters Tier3+ (BDs, cutscenes, ripperdoc, showers) and clears it when V returns to Tier1–2 (full gameplay or walkable scripted areas). Tier0 (undefined/uninitialized) is ignored to prevent mid-scene loading screens from clearing the block. The quest fact survives loading screens that wipe runtime state
- **Universal coverage** — instead of handling each scene type individually (braindance, ripperdoc, shower, etc.), a single tier-based guard catches all of them. The location callback, `OnGameAttached`, and `WardrobeLeaveHome` all check this flag and skip outfit changes when active

## ♻️ Internal Refactoring & Edge Case Fixes

A cleanup pass across the entire codebase — more robust and maintainable internals.

**Redundancy Removal:**
- **Extracted `ApplyLocationOrFallback` helper** — the location → context → outdoor fallback chain was duplicated in `OnWardrobeLocationChanged` and `WardrobeLeaveHome`. Now a single shared method in OutfitManager
- **Extracted `GuardedHomeSync` and `LoadNudePreserveTracking` helpers** — the guarded sync + nude loading pattern was duplicated across `UnequipWardrobeSet` and `OnQuestDisableWardrobeSetRequest`. Now two reusable methods in OutfitManager
- **Extracted `PickRandom` helper** — the random outfit selection pattern was repeated in three `Apply*` methods
- **Removed redundant `RefreshOutfitList` calls** — `HasOutfitForLocation` and `ApplyContextOutfit` no longer refresh the outfit list individually; `ApplyLocationOrFallback` refreshes once at the top for the entire chain

**Edge Case Fixes:**
- **Ripper flag stuck on fast travel** — if V fast-travels while in a ripperdoc workspot, `OnWorkspotFinishedEvent` may never fire, leaving `m_wardrobeInRipperWorkspot` stuck. The flag is now cleared on any location change (safe because ripper workspots don't trigger location changes)
- **Missing `m_wardrobeReady` guard on hangout callback** — `OnWardrobeHangoutEnded` could theoretically fire during initialization before the mod is ready. Now guarded
- **WardrobeLeaveHome missing location block guard** — `WardrobeLeaveHome` now checks `IsLocationBlocked` before applying outdoor outfit, preventing stale home-exit logic from firing during scene transitions
- **Ripper outfit corrupts tracking** — during ripperdoc sessions, game events (e.g. `UnequipWardrobeSet`) call `SyncCurrentOutfit`, which would pick up the temporary ripper outfit and overwrite the pre-ripper tracking. On workspot exit, the mod would try to "restore" the ripper outfit instead of the actual previous outfit. Fixed by guarding `SyncCurrentOutfit` to skip when `m_wardrobeInRipperWorkspot` is true
- **Stale location block on save/load** — `OnGameAttached` now checks the current PSM tier and clears `dw_block_location` only on explicit Tier1 (`== 1`), not Tier0 (`== 0`, undefined/uninitialized). Mid-scene loading screens trigger `OnGameAttached` with PSM not yet set up (tier 0), so using `<= 1` would incorrectly clear the block during BDs and cutscenes
- **Missing `IsBlocked()` guard on ripperdoc restore** — `OnWorkspotFinishedEvent` now checks `outfitSystem.IsBlocked()` before restoring the pre-ripper outfit, preventing quest outfit override on workspot exit

**Code Cleanup:**
- **Removed dead `m_wardrobeInCustomLocation` field** — leftover from an earlier design that was never cleaned up. Removed from Fields.reds and all assignment sites
- **Inverted empty if-block in `GetContextOutfits`** — replaced `if bad { } else if good { ... }` with `if good { ... }` for readability
- **Dynamic context key builder** — `ApplyContextOutfit` no longer hardcodes keyword order (`club`, `corpo`, `street`, `wild`). Now sorts the contexts array via bubble sort, making it future-proof for user-defined custom keywords in ContextMap.reds
- **Removed redundant `IsInBraindance` helper** — braindance suppression is now fully handled by the persisted `dw_block_location` scene tier guard. Combat and zone handlers use `IsLocationBlocked` instead, which covers all Tier3+ scenes (not just BDs)

## 🔧 Technical Details
- **`ApplyCombatOutfit`** — temporarily swaps `m_wardrobeOutfitList` to combat-only, then reuses the existing location → context fallback chain. Zero duplicated matching logic. Falls back to any combat outfit if no location/context match. Used by both combat entry and dangerous zone entry
- **Dangerous zone detection** — wraps the game's existing `OnZoneChange` callback on PlayerPuppet, which receives `SecurityAreaData` via a Variant listener on `PlayerStateMachine.SecurityZoneData`. `ESecurityAreaType.DANGEROUS` (value 3) triggers the combat outfit. `m_wardrobeInDangerousZone` flag tracks zone state separately from `m_wardrobeInCombat` so combat exit knows whether to restore
- **Scene tier location guard** — `OnSceneTierChange` wraps the game's existing PSM callback on PlayerPuppet. Tier3+ sets the persisted `dw_block_location` quest fact (survives loading screens that wipe runtime `@addField` values). Tier1–2 clears it (Tier0 is ignored — uninitialized PSM during mid-scene loads). The location callback, combat/zone handlers, `OnGameAttached`, and `WardrobeLeaveHome` all check this flag and skip outfit changes when active. Tier2 (walkable scripted areas like shops/clinics) is intentionally unblocked so location outfits apply naturally on entry
- **`ApplyLocationOrFallback`** — now filters out combat outfits before running the chain, preventing "combat street" from matching as a regular "street" outfit. Single entry point for the fallback chain: `RefreshOutfitList` → `HasOutfitForLocation` → `ApplyOutfitByLocation` / `HasContextOutfit` → `ApplyContextOutfit` / `ApplyOutfitByPrefix("outdoor")`. Called by both `OnWardrobeLocationChanged` and `WardrobeLeaveHome`
- **`GuardedHomeSync`** — wraps `SyncCurrentOutfit` with a home-only filter: syncs, checks `IsWearingPrefix("home")`, reverts tracking if the synced outfit isn't a home outfit. Prevents scene-triggered syncs from picking up nude/non-home outfits
- **`LoadNudePreserveTracking`** — saves `m_wardrobeLastOutfit`/`m_wardrobeLastContext`, loads nude (cached CName or by prefix), then restores tracking. Used by both `UnequipWardrobeSet` (drive scene) and `OnQuestDisableWardrobeSetRequest` (shower/sleep)
- **Ripper sync guard** — `SyncCurrentOutfit` returns immediately when `m_wardrobeInRipperWorkspot` is true, preventing game events during the ripperdoc session from overwriting the pre-ripper outfit tracking
- **Ripper flag reset** — `m_wardrobeInRipperWorkspot` cleared at the top of `OnWardrobeLocationChanged`, before any outfit logic runs. Location changes always fire on fast travel, so the flag cannot get stuck
- **Context key sort** — `ApplyContextOutfit` copies the contexts array, bubble-sorts it alphabetically via `StrCmp`, then joins with `+` and prefixes with `ctx:`. Custom keywords added to `ContextMap.reds` are automatically included in the dedup key
