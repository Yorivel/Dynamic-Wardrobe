# v1.4

## New Feature: Dogtown Hideout (Kress Street) Support
* **Phantom Liberty's Dogtown apartment** (`Dogtown_Hideout`) is now recognized as a home location
* Uses the same detection as vanilla apartments ‚Äî unique enum via the location callback, no configuration needed
* The engine only reports `Dogtown_Hideout` after the apartment is unlocked (quest-gated trigger volume), so no script-side quest fact check is required
* `Dogtown` added to `IsApartmentDistrict` for save/load fallback

## New Feature: River's Apartment (Trailer Park) Support
* **River's apartment** in the Badlands is now supported as a pre-configured custom apartment ‚Äî no setup needed
* **Quest-gated** ‚Äî only activates after completing "Following the River" (romance or friendship path)
* Uses OR gate logic: `sq029_river_lover` (romance) OR `sq029_done` / `sq029_stash` (friendship) ‚Äî any path unlocks it
* `hasShower = true` ‚Äî the apartment has a usable shower, so the shower deferral setting is respected
* Ships as a built-in entry in `CustomApartments.reds` and doubles as a practical example for users adding their own apartments

## New Feature: Custom Apartment Support

Dynamic Wardrobe now supports modded apartments. Register any apartment by adding its entrance coordinates to `Core/CustomApartments.reds` ‚Äî the mod handles the rest. River's apartment is included out of the box.

* **Simple config API** ‚Äî `DWApartment.Add(apartments, hasShower).Entrance(x, y, z)` with chained builder syntax
* **Multiple entrances** ‚Äî chain `.Entrance()` calls for apartments with more than one door
* **Quest gates** ‚Äî chain `.Gate("fact_name", minValue)` calls for story-locked apartments (AND logic ‚Äî all must pass), or `.OrGate()` for apartments unlockable via multiple quest paths (OR logic ‚Äî at least one must pass)
* **Shower deferral** ‚Äî `hasShower = true` respects the shower deferral setting; `hasShower = false` applies home outfit immediately on entry
* **Event-driven detection** ‚Äî uses SceneTier2 transitions + entrance proximity (no polling)
* **Persisted home state** ‚Äî `dw_is_at_home` now stores `2` for custom apartments (vs `1` for vanilla), enabling correct exit detection after save/load
* **Pre-configured** ‚Äî River's apartment (quest-gated with OR logic) ships ready to use
* **In-game helpers** ‚Äî CET console commands for coordinates and quest facts documented in the config file

## New Feature: Pinned Outfits (`!` Prefix)

Prefix any Equipment-EX outfit name with `!` to pin it ‚Äî Dynamic Wardrobe will skip all automatic location, context, combat, and save/load outfit swaps while the pinned outfit is equipped.

* **Stateless and disposable** ‚Äî no new settings, fields, or persisted data. The pin is detected by checking the currently equipped outfit name
* **Blocks** ‚Äî location changes, context matching, outdoor fallback, combat/danger zone entry, combat/zone exit restore, and home outfit restore on save/load
* **Plays normally** ‚Äî home outfit on apartment entry, ripper outfit on ripperdoc chair, nude during shower/sleep, and all scene restores. These consume the pin naturally (V's outfit changes, the `!` name is no longer equipped)
* **Matching exclusion** ‚Äî `!` outfits are filtered from all automatic matching pools. `!home cozy` will never be selected as a "home" outfit by the mod
* **Survives save/load** ‚Äî Equipment-EX restores the `!` outfit visually, and DW syncs its tracking on load to detect the pin before any outfit logic runs
* **Re-equip to re-pin** ‚Äî after a scene consumes the pin (e.g. shower ‚Üí nude ‚Üí home restore), simply re-equip the `!` outfit via EX to re-pin

## Bug Fix: Corpo Plaza Apartment Enum Mismatch
* **Problem**: Inside the Corpo Plaza apartment, the game can change the location enum to `CityCenter` or `Downtown` instead of the expected `CorpoPlaza`. This caused the mod's safety fallback to think V had left the apartment, triggering a location/context outfit swap (e.g. applying a "corpo" outfit) instead of keeping the home outfit.
* **Fix**: Added `CityCenter` and `Downtown` to `IsApartmentDistrict` so the safety fallback correctly recognizes V is still inside the Corpo Plaza apartment regardless of which enum the game reports.

## Bug Fix: Runtime Outfit Tracking Sync
* **Problem**: When V manually equipped an outfit via Equipment-EX, Dynamic Wardrobe's internal tracking didn't update until a game reload. This caused location/context outfits to override manual selections on the next location change.
* **Fix**: Added `SyncCurrentOutfit` (with combat guard) at the top of the location change callback. DW now syncs its tracking with EX's actual equipped outfit on every location change, catching manual outfit changes and refreshing the cached outfit list. In normal operation this adds just one lightweight `IsEquipped` check (early return when tracking is already correct).

## User Config Files

`Core/CustomApartments.reds` and `Core/ContextMap.reds` are user-editable config files. **Back up these files before updating** ‚Äî new versions will overwrite them.

## Technical Details
* **`OutfitManager.IsPinned`** ‚Äî checks if `m_wardrobeLastOutfit` starts with `!` via `StrBeginsWith`. No new fields or persisted facts
* **Pin guards** ‚Äî `ApplyLocationOrFallback`, `ApplyCombatOutfit`, combat/zone exit restore blocks, `OnWardrobeLocationChanged`, and `OnGameAttached` home restore all check `IsPinned` before loading outfits. State transitions (flags, facts) run untouched
* **Matching pool filter** ‚Äî `GetOutfitsByPrefix`, `ApplyOutfitByLocation`, and `GetContextOutfits` skip outfits starting with `!` so they can never be autonomously selected
* **Save/load sync** ‚Äî `OnGameAttached` calls `SyncCurrentOutfit` before any pin checks to populate `m_wardrobeLastOutfit` from EX's restored outfit (runtime `@addField` values are wiped on load)
* **Location change sync** ‚Äî `OnWardrobeLocationChanged` calls `SyncCurrentOutfit` (combat-guarded) before any logic to catch manual EX outfit changes and refresh the cached outfit list at runtime
* **`OnSceneTierChange` Tier2 entry** ‚Äî when SceneTier changes to 2 near a custom apartment entrance, `WardrobeEnterHome` fires. Tier1 return while `dw_is_at_home == 2` triggers `WardrobeLeaveHome`
* **`WardrobePersistHome`** ‚Äî now takes `Int32` instead of `Bool`: `0` = not home, `1` = vanilla apartment, `2` = custom apartment. Exit detection for custom apartments checks the persisted fact value instead of the runtime `m_wardrobeCurrentCustomApt` ref (which is wiped on load)
* **`DWApartmentDef`** ‚Äî data class with entrance coordinates, hasShower flag, and quest gates (AND + OR). Builder methods return `ref<DWApartmentDef>` for chaining
* **`HomeDetection.IsNearCustomApartment`** ‚Äî iterates custom apartment configs, checks entrance proximity (1.0 unit) and quest gates (both AND and OR), returns matching def or null
* **Safety fallback skip** ‚Äî location callback safety fallback (no Tier2, unknown district ‚Üí leave home) is skipped for custom apartments (`dw_is_at_home == 2`) since save/load may report unrelated location enums
