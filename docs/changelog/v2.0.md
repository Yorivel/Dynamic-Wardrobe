# v2.0

## Update Notice

This is a major update with new files and config changes. **Delete the old `Dynamic Wardrobe` folder entirely before installing v2.0.** If you've customized files in `UserConfig/`, back them up first — a new config file `UserConfig/VehicleContextMap.reds` is added in this version.

## New Feature: Vehicle Outfits

V can now wear different outfits while driving or riding. Three keywords control vehicle type:

* **`vehicle`** — any ride (cars and bikes)
* **`car`** — cars only
* **`bike`** — bikes only

Combine with context keywords for lore-appropriate driving looks: `car corpo`, `car street`, `bike wild`. Target specific vehicles by brand or model: `car caliburn`, `car herrera outlaw`, `bike kusanagi`.

* **Scoring tiers**: specific vehicle match > brand match > context match > type match > generic `vehicle`
* **Passenger mode**: outfit changes blocked as passenger by default (prevents swaps during quest rides). Configurable in Mod Settings
* **While mounted**: location, combat, weather, and shuffle changes are all paused
* **On dismount**: fresh location evaluation — the mod picks the best outfit for V's current location, never restores the old one
* **Decorative names**: extra words that aren't vehicle brands/models are ignored — `car casual`, `bike leather` work as expected
* **Customizable**: edit `UserConfig/VehicleContextMap.reds` to reclassify vehicles between corpo/street/wild

See [Vehicle Outfits](../vehicles.md) and [Vehicle Context Map](../vehicle-context-map.md) for details.

## New Feature: Weather & Time of Day

Add weather or time keywords to any outfit name as modifiers:

* **`rain`** / **`dry`** — react to rain conditions
* **`day`** / **`night`** — react to time of day (boundaries configurable in Mod Settings)

These act as **tiebreakers within the same tier** — `street rain` beats `street` when it's raining, but `street rain` never beats an exact location match like `afterlife`. Weather/time keywords work everywhere: outdoor, combat, club, and vehicle outfits.

* **Event-driven**: no polling — weather and time listeners fire only when conditions change
* **No re-roll on change**: if the weather changes but no weather-specific outfit exists for V's current area, nothing happens
* **Rain only**: weather detection is limited to rain/dry — fog, sandstorm, and other conditions can't be queried by Redscript mods

See [Weather & Time of Day](../weather-time.md) for details.

## New Feature: Outfit Shuffling

Automatic outfit rotation on a timer:

* **Context mode** — re-rolls within V's current location/context/outdoor pool
* **Chaos mode** — picks from all non-reserved outfits, ignoring location entirely
* **Timer**: configurable min/max interval (10s–900s), randomized each shuffle

Paused during combat, vehicles, home, clubs, scenes, and when pinned. See [Outfit Shuffling](../shuffling.md) for details.

## New Feature: Red Dirt Club Support

Red Dirt in Arroyo is now a supported club. Name outfits `club reddirt` or `club street` to match there. Two entrances are detected (front and back doors).

## New Feature: Partial Club Name Matching

Club names in outfit keywords now support partial matching — `club coyote` matches El Coyote Cojo, `club riot` matches Riot Club, `club lizzie` matches Lizzie's Bar. Any word of 4+ characters that's a substring of a known club name resolves automatically.

## New Feature: Combat Mode Setting

Control when combat outfits activate:

| Mode | What It Does |
|------|-------------|
| Combat + Danger Zones | Default — reacts to both fights and hostile area tags |
| Combat Only | Ignores danger zone tags (recommended if peaceful areas trigger combat outfits) |
| Danger Zones Only | Ignores combat PSM |
| Off | No combat outfits |

## New Feature: Home Outfits Toggle

Disable DW's home/nude/shower outfit management while keeping all other features active. Useful for compatibility with other apartment mods like Apartment Outfits.

## New Feature: Debug Overlay Overhaul

The F10 overlay is now a multi-page info panel:

* **Page 1 — Status**: Current location/vehicle, context, and wearing info
* **Page 2 — Matches**: Matching outfits with tier labels and winners
* **Page 3 — Tips**: Keywords that would match here + example outfit names
* **Page 4 — Facts**: Persisted quest facts with hold-to-reset
* **Page 5 — Examples**: Context-aware naming examples (on foot only)
* **Tap** to cycle pages, **hold** to dismiss

## Bug Fixes

* **Combat outfit on load**: Loading a save inside a hostile area no longer briefly shows a mixed outfit before combat gear kicks in
* **Club flickering**: Entering a club with no matching club outfits no longer causes rapid outfit cycling
* **Club-specific priority**: `club afterlife` now correctly takes priority over `club street` when at the Afterlife

## New Feature: Entrance Radius Setting

A new slider in Mod Settings > Troubleshooting lets you adjust how close V needs to be to a door for the mod to detect entry/exit (1.0–5.0 meters, default 1.0). Useful if apartment detection fails on some setups due to the game's trigger volume timing.

## Settings Reorganization

Settings are now organized into categories:

* **Behavior** — Home Outfits, Shower Deferral, Passenger Block, Combat Mode
* **Day / Night** — Day Start Hour, Night Start Hour
* **Shuffling** — Shuffle Mode, Min/Max Interval
* **Troubleshooting** — Entrance Radius

## Technical Details

* **Vehicle scoring** — `VehicleMatcher` splits the vehicle's TweakDB record ID (e.g. `Vehicle.v_sport2_quadra_type66_nomad_player`) into keywords and scores outfit names bidirectionally. Five tiers: record keyword match > brand match > type + context > type only > generic `vehicle`. Within each tier, more keyword hits = higher score, ties randomized
* **Weather/time as modifiers** — `WeatherTimeMatcher` classifies each outfit candidate as Enhanced (matching weather/time keywords), Neutral (no weather/time keywords), or Rejected (mismatching keywords). Enhanced always beats Neutral within the same location/context tier, but weather/time-only matches never beat a higher location tier. This is enforced at the matcher level, not the caller
* **`ShouldReEvaluate` guard** — weather/time changes only trigger outfit re-evaluation if the current outfit is Rejected (has non-matching keywords) or if the current outfit is Neutral and a better Enhanced outfit exists for the area. Prevents unnecessary re-rolls when the weather changes but no weather-specific outfits are configured
* **Time listeners** — `TimeSystem.RegisterIntervalListener` fires `DWDayStartEvent` / `DWNightStartEvent` at configurable hours, repeating every 24 in-game hours. Re-registered on settings change via `OnModSettingsChange`. Initial state bootstrapped from `TimeSystem.GetGameTime()` in `InitializeState`
* **Weather listener** — `WeatherScriptListener.OnRainIntensityTypeChanged` fires on rain state transitions. Dedup guard (`isRaining == m_wardrobeIsRaining`) prevents redundant re-evaluations when the intensity changes within the same rain/dry bucket
* **Shuffle timer** — `DelaySystem.DelayEvent` with randomized interval between configurable min/max seconds. Self-perpetuating: each `OnDWShuffleEvent` schedules the next timer before evaluating guards. Generation counter (`m_wardrobeShuffleGeneration`) rejects stale events from before BD transitions or save/load (where `@addField` delay IDs reset to 0)
* **Tier1 gate pattern** — all external triggers (weather, time, shuffle, vehicle) gate on `PSM SceneTier == 1` with fail-closed null check: `if !IsDefined(psmBB) { return; }`. The PSM blackboard can be null during Tier2 scenes — the old fail-open pattern (`if IsDefined(psmBB) { check tier }`) would skip the gate entirely on null, leaking outfit changes into BDs and shops
* **Dismount re-evaluation** — on vehicle exit, `m_wardrobeLastContext` is preserved (not cleared). Dedup guards in `ApplyOutfitByLocation` and `ApplyContextOutfit` compare the new context key against the old one and skip if unchanged. This prevents unnecessary re-rolls when V dismounts in the same area, while naturally re-evaluating if V drove to a new context
* **`dw_block_location` persistence** — location blocking uses a persisted quest fact instead of a runtime `@addField` flag because BD loading screens wipe all `@addField` values. Set on Tier4 entry, cleared on Tier1 return via `OnSceneTierChange`
