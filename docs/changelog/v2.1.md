# v2.1

## Update Notice

**Requires Equipment-EX v1.2.9 or later.** This version uses new EX APIs (`GetOutfitParts`) that don't exist in older versions — the script will fail to compile if your EX is outdated. Update Equipment-EX before installing v2.1.

## New Feature: Vehicle Accessories

Layer partial outfits — helmets, visors, gloves — on top of V's current outfit when entering a vehicle, without replacing the whole look. Prefix an outfit name with `(acc)` to mark it as an accessory layer instead of a full outfit.

* **Basic usage** — `(acc) bike helmet` layers a helmet when V gets on any bike
* **Vehicle targeting** — same scoring as full vehicle outfits: `(acc) car corpo gloves`, `(acc) bike kusanagi visor`
* **Full outfit priority** — if a full vehicle outfit matches (e.g. `bike nomad`), accessories are skipped entirely
* **Clean removal** — accessories are automatically stripped on dismount; V's previous outfit is fully restored
* **Pin safety** — `!` pinned outfits block both full vehicle outfits and accessories

See [Vehicle Outfits](../vehicles.md) for details.

## New Feature: Vehicle-in-Combat Setting

A new Mod Settings slider controls how vehicle outfits interact with combat:

| Mode | What It Does |
|------|-------------|
| 0 | Accessories layer on top of combat outfit |
| 1 | No accessories during combat (default) |
| 2 | Combat outfit overrides vehicle outfits entirely — no full outfit or accessories |

Found in **Mod Settings > Behavior > Vehicle Outfits in Combat**.

## New Feature: Outfit Families

Decorative keywords in outfit names now act as cross-context tiebreakers. If V is wearing `outdoor jacket day` and the time changes to night, the mod prefers `outdoor jacket night` over a random `outdoor casual night` — because they share the family keyword "jacket".

* **Opt-in by naming** — only activates if outfits share keywords that aren't consumed by any system. Users who don't use shared decorative keywords see no behavior change
* **Any shared keyword** — one matching family keyword is enough to connect outfits. `outdoor jacket leather day` matches `combat jacket` (shared: "jacket")
* **Cross-system** — works across all transitions: outdoor → combat, location → context, context → vehicle, day → night, etc.
* **Order-independent** — `outdoor jacket day` = `day jacket outdoor` = `jacket/outdoor/day`
* **Not a scoring boost** — family preference only runs after all scoring and filtering is complete. It's a tiebreaker within the final pool of equally-scored candidates. If no family match exists, falls back to random pick

**System keywords** (filtered out, never treated as family keywords): `outdoor`, `home`, `nude`, `ripper`, `combat`, `vehicle`, `car`, `bike`, `acc`, `corpo`, `street`, `wild`, `club`, all club names, `day`, `night`, `rain`, `dry`

See [Outfit Families](../priority-advanced.md#outfit-families) for details.

## Bug Fix: Outfit Restoration After Vehicle Rides

* **Problem**: Dismounting a vehicle would re-randomize V's outfit instead of restoring the exact outfit V was wearing before mounting. The vehicle mount overwrote the tracking fields, losing the original civilian outfit
* **Fix**: Dedicated pre-combat/vehicle fields now persist the civilian outfit across vehicle rides and combat. On dismount, the mod checks whether the saved outfit is still valid for V's current location — if yes, it restores exactly; if V drove to a different area, it picks the best match for the new location

## Bug Fix: Stale Combat Flags After Vehicle Dismount

* **Problem**: If combat ended while V was in a vehicle, the combat/danger zone flags stayed stuck — combat outfits stopped applying on the next fight. Present since v2.0
* **Fix**: Dismount now reads actual PSM state and syncs flags, clearing stale values left over from mid-ride state changes

## Bug Fix: Location Override While Mounted

* **Problem**: Driving through different location enums would trigger outfit changes mid-ride, overriding the vehicle outfit
* **Fix**: Location matching is now blocked while V is in a vehicle

## Technical Details

* **Accessory layering** — uses EX's `GetOutfitParts(name)` to read individual items from an `(acc)` outfit, then layers each via `EquipItem(tdbid)`. Tracked in `m_wardrobeActiveLayerParts`. No per-part `UnequipItem` needed on dismount — `LoadOutfit` on any restore path clears all slots
* **`(acc)` filtering** — accessory outfits return score 0 in `ScoreOutfit` (excluded from full vehicle matching) and are added to `IsReservedPrefix` (excluded from location/context matching). The `(acc)` prefix (5 chars including space) is stripped before scoring against vehicle keywords
* **`IsOutfitValidForLocation`** — dry-run validity check that runs through location keyword → context keyword → outdoor matching without calling `LoadOutfit`. Used by `RestorePreCombatOutfit` to decide whether to restore the exact outfit or re-evaluate for the new area
* **Pre-combat/vehicle preservation** — `m_wardrobePreCombatOutfit` / `m_wardrobePreCombatContext` fields saved on combat entry AND vehicle mount (only when tracking holds a civilian outfit). Restored on dismount/combat exit via `RestorePreCombatOutfit`
* **Mount/dismount sync** — `SyncCurrentOutfit` runs on both mount and dismount to catch manual EX outfit changes made while mounted
* **`PickPreferred`** — extracts family keywords from `m_wardrobeLastOutfit` using `GetFamilyKeywords` (filters via `IsSystemKeyword`), filters the candidate pool to outfits sharing any family keyword, picks random from filtered pool. Falls back to full pool if no family match. Called at all 8 outfit selection points (location, context, vehicle, accessories, outdoor, combat, club, prefix). Not called in shuffle chaos mode (intentionally fully random)
