# v2.1

## Update Notice

**Requires Equipment-EX v1.2.9 or later.** This version uses new EX APIs (`GetOutfitParts`) that don't exist in older versions — the script will fail to compile if your EX is outdated. Update Equipment-EX before installing v2.1.

## New Features

### Vehicle Accessories

Layer partial outfits — helmets, visors, gloves — on top of V's current outfit when entering a vehicle, without replacing the whole look. Prefix an outfit name with `(acc)` to mark it as an accessory layer instead of a full outfit.

* **Basic usage** — `(acc) bike helmet` layers a helmet when V gets on any bike
* **Vehicle targeting** — same scoring as full vehicle outfits: `(acc) car corpo gloves`, `(acc) bike kusanagi visor`
* **Full outfit priority** — if a full vehicle outfit matches (e.g. `bike nomad`), accessories are skipped entirely
* **Clean removal** — accessories are automatically stripped on dismount; V's previous outfit is fully restored
* **Pin safety** — `!` pinned outfits block both full vehicle outfits and accessories

See [Vehicle Outfits](../vehicles.md) for details.

### Outfit Restore on Exit

When V leaves an apartment or club, the mod now restores the exact outfit V was wearing before entering — instead of re-randomizing from the location pool. A new Mod Settings slider controls the behavior:

| Mode | What It Does |
|------|-------------|
| 0 | Restore outfit on both apartment and club exit (default) |
| 1 | Restore on club exit only; re-evaluate on apartment exit |
| 2 | Restore on apartment exit only; re-evaluate on club exit |
| 3 | Always re-evaluate (v2.0 behavior) |

* **Location validity check** — if V fast-traveled or the saved outfit doesn't match the current area, the mod falls through to normal location matching instead of blindly restoring
* **Pin safety** — `!` pinned outfits are saved and restored with the pin intact
* **Reserved outfit guard** — combat, home, and other reserved outfits are never saved as the pre-place outfit

Found in **Mod Settings > Behavior > Outfit Restore on Exit**.

### Vehicle-in-Combat Setting

A new Mod Settings slider controls how vehicle outfits interact with combat:

| Mode | What It Does |
|------|-------------|
| 0 | Accessories layer on top of combat outfit |
| 1 | No accessories during combat (default) |
| 2 | Combat outfit overrides vehicle outfits entirely — no full outfit or accessories |

Found in **Mod Settings > Behavior > Vehicle Outfits in Combat**.

### Outfit Families

Decorative keywords in outfit names now act as cross-context tiebreakers. If V is wearing `outdoor jacket day` and the time changes to night, the mod prefers `outdoor jacket night` over a random `outdoor casual night` — because they share the family keyword "jacket".

* **Opt-in by naming** — only activates if outfits share keywords that aren't consumed by any system. Users who don't use shared decorative keywords see no behavior change
* **Any shared keyword** — one matching family keyword is enough to connect outfits. `outdoor jacket leather day` matches `combat jacket` (shared: "jacket")
* **Cross-system** — works across all transitions: outdoor → combat, location → context, context → vehicle, day → night, etc.
* **Order-independent** — `outdoor jacket day` = `day jacket outdoor` = `jacket/outdoor/day`
* **Not a scoring boost** — family preference only runs after all scoring and filtering is complete. It's a tiebreaker within the final pool of equally-scored candidates. If no family match exists, falls back to random pick

**System keywords** (filtered out, never treated as family keywords): `outdoor`, `home`, `nude`, `ripper`, `combat`, `vehicle`, `car`, `bike`, `acc`, `corpo`, `street`, `wild`, `club`, all club names, `day`, `night`, `rain`, `dry`

See [Outfit Families](../priority-advanced.md#outfit-families) for details.

### Status Effect Entrance Detection

A new entrance type for places that lack both unique location enums and Tier 2 scene transitions — primarily modded clubs. Entry and exit are detected via status effect apply/remove events combined with proximity and district checks.

* **Zero overhead** — the handler early-outs on effect ID mismatch (one integer compare per status effect event). District checks and proximity math only run when the specific configured effect fires
* **Fully config-driven** — add `.StatusEffectEntrance(x, y, z, radius, t"EffectID")` to any place in `CustomPlaces.reds`
* **Gomorrah Night Club** — first supported modded club ([Nexus link](https://www.nexusmods.com/cyberpunk2077/mods/23350)). Detected via `GameplayRestriction.NoCombat` effect. Context: `street` + `club`. Outfit targeting: `club gomorrah` or partial like `club gomor`. No-op if the mod isn't installed

## New Places & Apartments

### New Clubs: 7th Hell, Electric Orgasm, Dicky Twister, Empathy, Ho-Oh, The Moth, Gomorrah

Added seven new clubs as supported places. Six use Tier 2 entrance detection since the game doesn't report unique location enums for these venues. Gomorrah uses status effect detection (see below).

* **7th Hell** — at CityCenter/Downtown border. Context: `street` + `club`. Outfit targeting: `club 7thhell`
* **Electric Orgasm** (Dino Dinovic's bar) — Downtown/CityCenter. Context: `street` + `club`. Outfit targeting: `club electricorgasm` or partial like `club electric`
* **Dicky Twister** — Vista del Rey. Context: `street` + `club`. Four Tier 2 entrances (front door + three interior doors). Outfit targeting: `club dickytwister` or partial like `club dicky`
* **Empathy** — CityCenter/Downtown. Context: `corpo` + `club`. Outfit targeting: `club empathy`
* **Ho-Oh** — Kabuki. Context: `street` + `club`. Two Tier 2 entrances (front door + upstairs). Outfit targeting: `club hooh`
* **The Moth** — Dogtown. Context: `wild` + `club`. Outfit targeting: `club themoth` or partial like `club moth`
* **Gomorrah Night Club** (modded — [Nexus link](https://www.nexusmods.com/cyberpunk2077/mods/23350)) — Little China. Context: `street` + `club`. Detected via `NoCombat` status effect. Outfit targeting: `club gomorrah` or partial like `club gomor`. No-op if the mod isn't installed

### Panam's Bed (New Aldecaldos Camp)

V's home outfit now applies when using Panam's bed at the new Aldecaldos camp. Gated by the `sq027_panam_lover` quest fact (Panam romance path). The outfit restores when V stands up — the restore is deferred until V is actually out of bed, not when the skip-time UI closes.

### New Custom Apartment: Dogtown Hideout (Modded)

Added support for the [Dogtown Hideout](https://www.nexusmods.com/cyberpunk2077/mods/12008) custom apartment mod. Two entrances (main door + trapdoor). No-op if the mod isn't installed.

## Bug Fixes

### Outfit Restoration After Vehicle Rides

* **Problem**: Dismounting a vehicle would re-randomize V's outfit instead of restoring the exact outfit V was wearing before mounting. The vehicle mount overwrote the tracking fields, losing the original civilian outfit
* **Fix**: Dedicated pre-combat/vehicle fields now persist the civilian outfit across vehicle rides and combat. On dismount, the mod checks whether the saved outfit is still valid for V's current location — if yes, it restores exactly; if V drove to a different area, it picks the best match for the new location

### Stale Combat Flags After Vehicle Dismount

* **Problem**: If combat ended while V was in a vehicle, the combat/danger zone flags stayed stuck — combat outfits stopped applying on the next fight. Present since v2.0
* **Fix**: Dismount now reads actual PSM state and syncs flags, clearing stale values left over from mid-ride state changes

### Combat Outfit Swap at Clubs

* **Problem**: Entering combat at a club (bar fights, etc.) would trigger a combat outfit swap, causing outfit visuals to merge together
* **Fix**: Combat and danger zone handlers now skip outfit swaps when V is at a club/place, same as the existing home skip

### Stale Home/Place Flags on Vehicle Mount

* **Problem**: If place or home exit detection missed (e.g. fast travel edge case, missed Tier1 transition), the stale flag would permanently block vehicle outfits and location matching
* **Fix**: Vehicle mount and dismount now clear stale home/place flags as a safety fallback — V can never be inside a home or club while riding a vehicle

## Improvements

### Club Context Decoupled from District

Club synthetic enum names no longer include the parent district prefix when the club's vibe doesn't match the district's geographic context. This prevents unwanted context inheritance — e.g. 7th Hell (a street-vibe club in the corpo Downtown district) no longer pulls in `corpo` context. Heavy Hearts (a corpo-vibe club in Dogtown) no longer pulls in `wild` context.

### Debug Overlay Shows Synthetic Place Names

The F10 debug overlay now displays the mod's synthetic enum name (e.g. `7THHELL`) when V is inside a detected place, instead of showing only the parent district the game reports. All overlay pages (status, matches, tips, examples) resolve the synthetic name. Makes it much easier to verify place detection is working and to find the correct keywords for outfit naming.

## Technical Details

* **Accessory layering** — uses EX's `GetOutfitParts(name)` to read individual items from an `(acc)` outfit, then layers each via `EquipItem(tdbid)`. Tracked in `m_wardrobeActiveLayerParts`. No per-part `UnequipItem` needed on dismount — `LoadOutfit` on any restore path clears all slots
* **`(acc)` filtering** — accessory outfits return score 0 in `ScoreOutfit` (excluded from full vehicle matching) and are added to `IsReservedPrefix` (excluded from location/context matching). The `(acc)` prefix (5 chars) is stripped before scoring against vehicle keywords — any trailing space is consumed by the keyword splitter as a delimiter
* **`IsOutfitValidForLocation`** — dry-run validity check that runs through location keyword → context keyword → outdoor matching without calling `LoadOutfit`. Used by `RestorePreCombatOutfit` to decide whether to restore the exact outfit or re-evaluate for the new area
* **Pre-combat/vehicle preservation** — `m_wardrobePreCombatOutfit` / `m_wardrobePreCombatContext` fields saved on combat entry AND vehicle mount (only when tracking holds a civilian outfit). Restored on dismount/combat exit via `RestorePreCombatOutfit`
* **Mount/dismount sync** — `SyncCurrentOutfit` runs on both mount and dismount to catch manual EX outfit changes made while mounted
* **`PickPreferred`** — extracts family keywords from `m_wardrobeLastOutfit` using `GetFamilyKeywords` (filters via `IsSystemKeyword`), filters the candidate pool to outfits sharing any family keyword, picks random from filtered pool. Falls back to full pool if no family match. Called at all 8 outfit selection points (location, context, vehicle, accessories, outdoor, combat, club, prefix). Not called in shuffle chaos mode (intentionally fully random)
* **Pre-place outfit restore** — `m_wardrobePrePlaceOutfit` / `m_wardrobePrePlaceContext` saved on home/club entry (gated by `outfitRestore` setting and `IsReservedPrefix`). On exit, validated via `IsOutfitValidForLocation` before restoring — falls through to `ApplyLocationOrFallback` if invalid or if the setting disables restore for that place type
* **Status effect detection** — `OnStatusEffectApplied`/`OnStatusEffectRemoved` wraps on `PlayerPuppet`. Fast gate via `IsRelevantEffect(effectID)` (single TweakDBID compare) before any place iteration. On match: district gate → proximity check → `WardrobeEnterPlace`. On removal: persisted place ID lookup → effect match → `WardrobeLeavePlace`
* **Panam's bed** — detected via Tier4 entry + proximity to bed coordinates (6.0 radius) + `sq027_panam_lover` quest fact. Reuses ripperdoc workspot flag (`m_wardrobeInRipperWorkspot`) — same save/restore pattern but applies "home" prefix. Restore deferred from `OnWorkspotFinishedEvent` to `OnSceneTierChange(1)` because the workspot finishes while V is still lying down (skip-time UI closes before the stand-up animation)
* **Vehicle mount/dismount safety** — stale `m_wardrobeIsAtHome`/`m_wardrobeIsAtPlace` flags cleared on mount and dismount, with `WardrobePersistHome(0)` to sync the quest fact. Replaces the old early-return guard that silently skipped vehicle outfit application
